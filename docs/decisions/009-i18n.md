# ADR 009: Internationalization of canonical data formats

**Status:** Accepted
**Date:** 2026-04-22

## Context
David's network includes contacts in the UK, Canada, Ireland, and the US. Phone numbers entered without a consistent canonical format would produce silent duplicates (a contact entered as "020 7946 0958" from the UK would not match "+44 20 7946 0958" entered later). Names, dates, and addresses also have varying conventions.

## Decision
- **Phone numbers: E.164 storage, library-parsed.**
  - Use `libphonenumber-js` to parse all phone input.
  - Store the result in E.164 format (`+14155551234`, `+442079460958`, etc.).
  - Each contact has an optional `defaultCountry: string` field (ISO 3166-1 alpha-2 code: `US`, `GB`, `CA`, `IE`, etc.). When a phone number is entered without a leading `+`, the parser uses the contact's `defaultCountry` to disambiguate.
  - If no contact `defaultCountry` is set, fall back to `DEFAULT_COUNTRY` from `.env` (default `US`).
  - For display, format E.164 back into the contact's country-local convention at render time.
  - Invalid phone numbers fail Zod validation with a clear error.

- **Names: preserve as entered, normalize whitespace only.**
  - Trim leading/trailing whitespace; collapse internal runs of whitespace to single spaces.
  - Preserve diacritics exactly as entered. Search uses case-folded + diacritic-folded comparison but storage is unchanged.
  - Optional `preferredName` field for the name David actually uses in communication (e.g., "Greg" when legal name is "Gregory").

- **Email: lowercase the domain portion, preserve the local part.**
  - `David.Jones@Example.COM` stored as `David.Jones@example.com`. Local part (before `@`) is case-sensitive per RFC; domain is not.

- **URLs: lowercase hostname, strip trailing slash, preserve path case.**

- **Addresses: freeform text, no parsing.**
  - Not used for functional features (no mapping, no mail merge). No value in parsing into components.

- **Dates and times: UTC ISO 8601 with Z suffix for storage, local timezone for display.**
  - Already covered by ADR 006 but reaffirmed here.

- **Currency: not in the data model.**

## Consequences
**Easier:** no silent duplicates. Search finds contacts regardless of how their phone was originally typed. International contacts are first-class citizens, not edge cases.

**Harder:** one extra library dependency; one extra step in input validation. Small cost.

**New risks:** `libphonenumber-js` metadata is large (~150KB gzipped in minimal build). Acceptable for a desktop-class app.

## Alternatives considered
- **Store phone numbers as entered:** leads to duplicates and broken search across locales.
- **US-only:** doesn't fit David's actual network.
- **Parse addresses into components (street, city, postcode):** no feature needs it; over-engineering.
